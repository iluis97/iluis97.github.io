MySQL 性能优化

发表于2025-05-17|更新于2025-06-08|MySQL
|阅读量:

设计索引原则

原则说明
1. 优先选择高选择性字段建索引选择性 = 不同值的数量 / 总记录数。选择性越高（越稀疏），过滤效果越好。例如：身份证号、手机号比性别更适合建索引。
2. 频繁作为查询条件的字段应建索引对 WHERE 子句中经常使用的字段建索引可显著提升查询效率。
3. 出现在 JOIN / ORDER BY / GROUP BY 的字段需特别关注JOIN 的连接字段、排序字段、分组字段需要考虑建立索引以避免全表扫描、文件排序等性能瓶颈。
4. 覆盖索引优先通过建立包含查询字段的联合索引，可以避免回表，提高查询性能。
5. 索引列顺序要符合查询习惯（最左前缀原则）复合索引遵循最左匹配原则，索引列的顺序要匹配常用查询的字段顺序。
6. 控制索引数量，避免冗余每个索引会占用空间、影响写性能（INSERT/UPDATE/DELETE 时都要维护索引）。要权衡读取与写入性能。
7. 避免在频繁更新的字段上建立索引频繁变更的字段若被索引，会带来较大的维护成本。
8. 避免在表达式、函数结果上建索引如 WHERE YEAR(birthday) = 1990 这样的查询无法命中索引。应改写为 birthday BETWEEN '1990-01-01' AND '1990-12-31'。

注：

  • 选择性越高（基数越高），索引过滤数据的能力越强，效果越好
  • 选择合适的索引类型、合适的索引长度
  • 创建过多索引，占用磁盘空间，影响写入性能，因为数据新增、删除也要维护索引
  • 数据量大小，小表无需索引（全表扫描反而更快）
  • 主键 / 唯一约束设计，主键默认是聚簇索引（InnoDB），要选择稳定、唯一、短小的字段作为主键。
  • DBA 工具或慢查询日志发现热点查询，针对性优化
  • 设计复合索引时，要考虑“选择性最高的字段靠前”，以提升过滤效率
  • 删除冗余索引，减少优化器的选择干扰

区分度不高的字段是否该创建索引

推荐场景：

  • 可与高区分度字段构成联合索引，可以覆盖索引
  • 可用于排序 / 分组 / 连接
  • 用于频繁过滤的字段，如is_deleted = 1，schedule_done = 0 若系统频繁查询未删除数据或查询未运行的定时任务，建索引可使查询更快。

不推荐场景：

  • 单独字段、选择性极低、数据量小、参与的查询极少
  • 低区分度单字段索引 + 查询结果占数据大多数：索引命中后仍需大量回表，性能无提升甚至更差。

性能优化

like模糊查询如何优化

  • 避免开头使用 %
  • 利用覆盖索引
  • 反向索引（逆序存储字段），需额外存储字段和维护
  • 外部搜索引擎（Elasticsearch）

反向索引实现

如果必须用join，可以从哪些方面优化

  1. 联结条件使用索引
  2. 小表驱动大表
  3. 数据过滤后再join
  4. 优先使用内连接（仅保留共有数据）

file sort 优化

优化点说明示例

避免多列排序导致 File Sort多列排序时索引必须覆盖所有排序字段及顺序ORDER BY col1 ASC, col2 DESC 必须对应索引顺序

减少排序数据量加限制条件减少待排序行数增加 WHERE 限制或 LIMIT

分页时避免大 OFFSET大 OFFSET 导致大量扫描和排序用索引条件分页代替 OFFSET
利用覆盖索引避免回表只读取索引列避免访问数据行，提升效率选择索引列作为查询字段

执行计划

  • 执行计划：是 MySQL 优化器根据查询语句、表结构、索引、统计信息等因素，计算出的它认为最优的查询执行路径的详细描述。
  • 优化器：MySQL 内置的组件，负责在众多可能的执行路径中选择它预估成本最低的一个。
  • 成本：优化器基于 CPU、内存、磁盘 I/O 等操作的预估开销来计算不同执行路径的成本。

部分字段汇总

id

标识 SELECT 查询的序列号

  • 如果查询包含子查询或 UNION，id 会反映执行顺序。id 越大，执行优先级越高；
  • id 相同，执行顺序由上到下。
  • id 为 NULL 表示是 UNION 结果或派生表。

rows (预估行数 - 重要！)

MySQL 优化器预估执行该步骤需要扫描的行数（不是精确值，基于统计信息）。

这个值对于判断效率非常关键。理想情况下，这个值应尽可能小。与 filtered 结合使用。

filtered (百分比 - 重要！)

表示存储引擎返回的数据在 rows 行中，经过 WHERE 条件过滤后，预估剩余行数的百分比。值范围 0.00 - 100.00。

例如：

  • rows=1000, filtered=10.00，表示优化器预估存储引擎返回 1000 行，在 Server 层应用 WHERE 条件后大约剩下 100 行。

  • EXPLAIN FORMAT=JSON 会直接给出 filtered 对应的实际预估行数 (rows_examined_per_scan * filtered / 100)。

id 与 select_type 的配合

  • 查询的类型
select_type场景
SIMPLE单表或未嵌套查询、无子查询
PRIMARY最外层 SELECT
SUBQUERY子查询中 SELECT
DERIVEDFROM 子句中的子查询（派生表）
UNIONUNION 中的第二个及以后 SELECT
UNION RESULTUNION 的结果
DEPENDENT依赖外层结果的子查询
DEPENDENT SUBQUERY依赖于外层查询结果的子查询（通常性能较差）
MATERIALIZED对子查询物化处理，将子查询结果物化为临时表以提高性能

拓展：

MySQL 8.0.16+ 开始，在执行计划中正式引入 MATERIALIZED 作为 select_type 的一种类型，用于标识：

  • 子查询被优化为物化（生成临时表）。
  • 常见于 IN (SELECT …) 或 EXISTS 的场景中。

type — 表访问方式的重要排序

  • MySQL 在表中查找行的方式

性能从最优到最差大致为：

type含义及性能解读
system表只有一行（常用于 DUAL）
const主键或唯一索引匹配常量值，最多一行
eq_ref连接操作中，使用唯一索引匹配（如主键连接）
ref非唯一索引的等值匹配， 可能匹配多个值
index_merge多索引合并，非唯一索引匹配多个值
range范围扫描索引（如 BETWEEN、>, <）
index全索引扫描（未使用 WHERE 过滤）
ALL全表扫描（慢，需优化）

拓展

  • eq_ref： 在连接查询中，对于前表的每一行，从当前表中使用主键或唯一非空索引查找唯一一行。常见于 PRIMARY KEY 或 UNIQUE NOT
    NULL 索引的等值连接 (table.column = other_table.column)。

  • ref： 使用非唯一索引进行等值查找 (key_column = expr)，可能返回多行。

  • ref_or_null： 类似 ref，但额外处理 IS NULL。

  • range： 使用索引检索给定范围的行 (BETWEEN, IN, >, < 等)。

  • index： 全索引扫描。扫描整个索引树（比全表扫描快，因为索引文件通常比数据文件小）。通常发生在只需要从索引中获取数据，或者需要按索引顺序读取时。

  • ALL： 全表扫描。遍历整张表查找匹配行。最差情况，需极力避免，尤其在大表上。如果 WHERE 条件没有可用索引，或者优化器判断全表扫描成本更低（表很小）时会发生。

key_len

使用的索引的长度（字节数）。可以判断使用了复合索引的多少部分（前缀）。

字段类型字节数

可以使用 EXPLAIN FORMAT=JSON 获取 key_len 的实际值。

结果（关注 key、key_len、 key_parts）

ref

用来与索引列进行等值匹配的字段或常量，简单说：MySQL 用哪个值（或字段）在索引里“查找”

格式如 const, func, database.table.column, NULL。

ref 值类型含义示例

Extra 字段常见信息说明

Extra 内容含义与优化建议
Using where使用了 WHERE 过滤条件，存储引擎返回数据后在Server层过滤
Using index覆盖索引，无需回表，性能好
Using index conditionMySQL 5.6+ 的 ICP（索引条件下推）优化

Materialize子查询被物化为临时表
Optimize table表优化提示

Using index condition（索引条件下推）

工作流程对比

阶段无 ICP有 ICP（Using index condition）
索引扫描查找到匹配的索引项查找到匹配索引项
回表回表读取整行数据存储引擎提前根据条件判断是否满足，再决定是否回表
Server 层过滤判断 WHERE 条件是否满足多数数据已在存储引擎中过滤

没有 ICP 时问题在于：大量无效数据也被回表，性能下降。

ICP 总结一句话

  • 将部分 WHERE 逻辑从Server 层“下沉”到存储引擎层，提前过滤无效数据，减少回表，提高性能。
  • 是非覆盖索引查询中非常重要的性能优化机制。

使用场景

  • 满足以下条件之一，ICP 才可能被启用
条件描述
有范围查询如 IN, BETWEEN, >, <, LIKE 等
索引字段部分匹配查询条件中有部分字段使用了索引中的一部分
使用了复合索引但不是最左前缀全匹配例如 (col1, col2)，查询 WHERE col1 = ? AND col2 > ?
非覆盖索引查询如果是 SELECT * 或查询字段未被索引覆盖

Using sort_union(uniq_username,PRIMARY); Using where

这是 index_merge 策略 的一种具体实现形式。 表示 MySQL 分别使用 uniq_username 和 PRIMARY 两个索引进行索引扫描；

然后对扫描结果（主键 ID 列）进行 排序合并（sort union）；最终再根据这些主键 ID 去主表取出完整的行数据。

这种策略常用于：

  • WHERE 条件包含多个字段的 OR

  • 每个字段上都有独立索引

  • 且字段之间无法合并为一个复合索引

  • 和 Using union(…) 不同，Using sort_union(…) 会对每个索引返回的结果先排序再合并，适用于较小的结果集。

为什么是 sort_union 而不是 union？

  • union(PRIMARY, uniq_username)：表示直接合并两个索引的结果集（无序），适合结果集较大且无排序要求。

  • sort_union(…)：表示两个索引各自结果排完序后再合并，适合较小结果集，合并效率更高。

MySQL 会根据索引返回数据的量和散布程度，动态选择 sort_union 或 union。

分析步骤建议

警惕性能杀手

  • type: ALL (全表扫描) - 大表上必须避免。
  • type: index (全索引扫描) - 虽然比 ALL 好，但扫描大量索引行也可能很慢。
  • Using temporary (创建临时表) - 特别是磁盘临时表。
  • Using filesort (文件排序) - 特别是大结果集排序。
  • Using join buffer - 可能意味着连接缺少有效索引。
  • 高 rows 值 + 低 filtered 值 - 表示大量数据被读取但大部分被丢弃。
  • 看 key 和 key_len：是否用了合适索引。
  • 隐式转换：保证比较时数据类型一致

优化建议

  • 追求覆盖索引 (Using index)： 尽可能让索引包含查询所需的所有列，减少回表操作。

  • 理解索引条件下推(Using index condition)：这是好的优化标志。

  • 利用 EXPLAIN FORMAT=JSON 深入分析成本： 理解优化器为何选择该计划。

  • 保持统计信息更新 (ANALYZE TABLE)： 确保优化器估算准确。

  • 合理创建索引：基于查询的 WHERE, JOIN, ORDER BY, GROUP BY 子句创建合适的单列索引、联合索引（注意顺序！）。

  • 复合索引顺序原则：区分度高的列在前，范围查询列在后。

  • 关注连接顺序：通常让结果集小的表作为驱动表，选择性高的条件先过滤。

  • 任何索引添加都应通过EXPLAIN确认效果。

  • 善用 FORCE INDEX（是补救措施，不是优化首选，不代表 MySQL 一定会使用它）。

执行计划的局限性

  • 预估而非实测： rows 和 filtered 是基于统计信息的估计值，可能与实际情况有偏差。统计信息过时会导致预估不准。

  • 不包含实际执行细节：不显示实际执行时间、锁等待、临时表是否在磁盘、具体排序算法等。
    需结合 Performance Schema (SELECT * FROM performance_schema.events_statements_history_long ...)
    或慢查询日志分析实际执行。

  • 不显示所有优化细节： 一些内部优化（如常量传播）在计划中可能不明显。

  • 计划可能变化： 数据量变化、统计信息更新、MySQL 版本升级、配置参数改变都可能导致同一查询的执行计划改变。
