MYSQL 架构

连接层（Connection Layer / Client Connectors）

核心目标： 建立、管理和保护客户端与MySQL服务器之间的安全通道。

  • 连接管理与线程池
  • 身份认证与安全
  • 权限初步验证
  • 会话管理
  • 通信协议支持

服务层（Server Layer / SQL Layer）

核心目标： 理解用户请求（SQL）、检查权限、找出最高效的执行方式（优化）、调用存储引擎执行、管理部分服务级别的功能和元数据。

  • SQL接口（SQL Interface）: 接收客户端通过连接层发送过来的SQL语句或其他命令。
  • 解析器（Parser）: 对SQL语句进行词法分析和语法分析。检查SQL语法是否正确，并将其解析成MySQL内部能理解的语法结构树（Abstract
    Syntax Tree, AST）。
  • 优化器（Optimizer）: 这是服务层的核心组件。基于解析树、表结构、索引信息、系统统计数据和配置参数
  • 查询缓存（Query Cache - Removed in MySQL 8.0）
  • 内置函数与存储过程/触发器: 管理、解释和执行内置的SQL函数（如 SUM(), MAX()）、存储过程、用户定义函数（UDF）、触发器和视图的定义与执行逻辑。
  • 权限管理（高级）:
    在执行优化后的操作之前，检查用户是否有权限执行该SQL语句（如SELECT、UPDATE、DELETE权限，特定表的权限等）。权限信息通常在
    mysql 数据库中。
  • 缓存与缓冲区管理: 管理表定义信息缓存（table_definition_cache）、权限缓存等一些元数据缓存。
  • Binlog（二进制日志）管理: 服务层负责将已提交的事务记录到Binlog中（与存储引擎无关的日志，用于主从复制和特定场景的数据恢复）。

解析器

一定很熟悉

优化器

  • 制定一个或多个可能的执行计划。
  • 预估每个执行计划的成本（I/O开销、CPU开销等）。
  • 选择它认为成本最低的执行计划。优化策略包括索引选择、连接顺序优化、子查询优化等。

执行器

  • 要判断对这个表是否有执行查询的权限，若无则返回没有权限的错误
  • 有权限，就打开表继续执行。打开表的时候，执行器就会根据表的引擎定义，去使用这个引擎提供的接口。
  • 取表的数据，判断逻辑，不满足忽略，满足将这行存在结果集中，直到取到这个表的最后一行。

存储引擎层（Storage Engine Layer）

核心目标： 负责数据的物理存储、提供高效的读写访问接口、保证数据一致性和持久性（如果支持事务）、通过锁或MVCC管理并发访问。其具体行为因引擎而异。

  • 核心数据存储与检索
  • 事务管理（如果引擎支持）: ACID实现、MVCC（多版本并发控制）
  • 行级/表级锁
  • 索引实现
  • 故障恢复
  • 插件式架构
  • 缓冲管理：
    • InnoDB Buffer Pool: 核心的缓存，将表数据和索引页缓存在内存中，大大减少磁盘I/O。
    • MyISAM Key Buffer: 只缓存索引，数据依赖操作系统缓存。

回表操作

回表操作（row lookup / table lookup）在 MySQL 中发生在 存储引擎层（Storage Engine Layer），

当 MySQL 查询使用 非覆盖索引 时：

  • 先在二级索引 中找到满足条件的主键（rowid）
  • 然后再去 主键聚簇索引 中，通过 rowid 查找实际数据行

这个从索引页跳转到数据页的过程，就是所谓的 回表（Row Lookup）。

事务

什么是事务（Transaction）

事务是一组操作的集合，要么全部成功，要么全部失败，具有以下 4 大特性（ACID）

特性描述关键目标
原子性（Atomicity）事务是最小的执行单元，要么全部执行成功，要么全部失败回滚。避免部分提交
一致性（Consistency）事务执行前后，数据库必须处于一致状态， 数据必须满足所有业务规则、约束、外键、触发器等。数据逻辑正确性
隔离性（Isolation）多个事务并发执行时，互不干扰控制并发影响
持久性（Durability）事务一旦提交，对数据库的修改就是永久性的，即使系统宕机也不会丢失。故障后数据不丢失

事务隔离级别（Isolation Levels）

隔离级别脏读不可重复读幻读实现机制
READ UNCOMMITTED❌❌❌无锁，直接读最新数据
READ COMMITTED (RC)✅❌❌MVCC（每次读最新快照）
REPEATABLE READ (RR)✅✅⚠️MVCC（事务开始时快照） + Next-Key Lock
SERIALIZABLE✅✅✅全表锁 + 读锁

RR 级别通过 Next-Key Lock 解决了“写操作导致的幻读”，但“读操作幻读”在特定场景下仍可能发生

注：

  • SERIALIZABLE: 所有 SELECT 语句隐式转换为 SELECT … FOR SHARE（加共享锁）。
  • Next-Key Lock = Record Lock（记录锁）+ Gap Lock（间隙锁）

常见问题：

  • 脏读：读取到其他事务未提交的修改
  • 不可重复读：同一事务内多次读取结果不一致（因其他事务提交修改）
  • 幻读：同一事务内范围查询结果集不一致（因其他事务插入新数据）

官方文档：https://dev.mysql.com/doc/refman/8.0/en/innodb-transaction-isolation-levels.html

特性Redo LogUndo LogBinlog

Redo Log (重做日志)

目的：确保事务的持久性（Durability），redo log 是 InnoDB 引擎特有的日志。

物理级日志（记录数据页的变更）, 如”哪一页 Page 的哪一个偏移 offset 改成了什么值”。

关键特性：

  • 物理日志：记录数据页的字节级修改，物理页层面的二进制日志，非文本可读，需要通过工具解析（如 innodb_ruby）。
  • 循环写入：固定大小文件组（如 ib_logfile0, ib_logfile1），写满后覆盖旧日志。
  • 强制刷盘：事务提交时，Redo Log 必须刷盘（通过 innodb_flush_log_at_trx_commit=1 保证）。
  • 崩溃恢复：数据库重启后，通过重放 Redo Log 恢复已提交但未刷盘的数据。
  • 写入优化：将随机磁盘 I/O（数据页修改）转换为顺序 I/O（日志写入）。

特点：

特性内容
写入时机写入内存日志缓冲区 redo log buffer，再异步刷盘
写入机制Write-Ahead Logging（先写日志再写数据页）
数据结构固定大小（circular buffer 循环写入）

生命周期：

  • 事务提交 → Redo Log 刷盘 → 后台线程将脏页刷盘 → Redo Log 可覆盖

有了 redo log，InnoDB 就可以保证即使数据库发生异常重启，之前提交的记录都不会丢失，这个能力称为crash-safe。

Undo Log (回滚日志)

目的：

  • 提供事务回滚能力
  • 支持 MVCC 快照读（读取旧版本）
  • 每条记录都保存“被更新前的旧值”

关键特性：

  • MVCC 支持：为其他事务提供历史版本数据，实现非锁定读（如 SELECT 不阻塞写）。
  • 逻辑日志：记录逆向操作（如 INSERT 对应 DELETE，UPDATE 记录旧值）。
  • 链式存储：存储在Undo Tablespaces（系统表空间或独立文件）。
  • 延迟清理：事务提交后，Undo Log 不会立刻删除（供 MVCC 使用），由后台 Purge 线程回收。
  • 事务回滚：版本链遍历，如果当前版本不可见，则沿着 DB_ROLL_PTR 指向的旧版本依次判断。
  • 崩溃恢复：协助回滚未提交的事务。

生命周期：

  • 事务开始 → 修改前写入 Undo Log → 事务提交/回滚 → Purge 线程清理
  • Undo log 被提交且不再被任何活动事务需要后才能被删除。

Binlog (归档日志)

binlog 是 MySQL 的 Server 层实现的，所有引擎都可以使用。

作用

  • MySQL 层面的事务提交日志，记录 SQL 逻辑事件
  • 支持 主从复制、数据恢复、闪回

核心功能：

  • 主从复制：从库通过重放 Binlog 实现与主库数据同步。
  • 数据恢复：配合全量备份 + Binlog 恢复到任意历史时间点。
  • 审计：记录所有数据变更操作（DDL/DML）。

关键特性：

  • 逻辑日志：记录SQL 语句（Statement 格式）或行数据变更（Row 格式）。
  • 追加写入：文件持续增长（如 binlog.000001）写到一定大小后会切换到下一个，通过 expire_logs_days 自动清理。
  • 异步刷盘：事务提交时不强制立即刷盘（由 sync_binlog 参数控制频率）。

生命周期：

  • 事务提交 → 写入 Binlog Buffer → 刷盘到 Binlog 文件 → 复制到从库/用于恢复

数据样式（根据 binlog_format 而异）

STATEMENT：记录原始的SQL语句。

ROW (推荐)：记录每一行数据修改前后的状态（默认且最安全）。

MIXED：混合模式，大部分用STATEMENT，特定情况（如非确定性函数）自动转ROW。

RC隔离级别 + Statement模式 问题

mysql不允许，会出现一些数据不一致、主从不一致、复制异常等问题，这是生产环境中应极力避免的组合。

幻读问题
项说明
RC（Read Committed）每次读取都能看到已经提交的数据，不具备快照一致性
Statement Binlog 模式Binlog 中记录的是 SQL 语句文本，重放时会重新执行一遍 SQL
核心冲突点在 RC 下，相同 SQL 可能在不同时间读取到不同数据，而 Statement 模式会复制并重新执行 SQL，造成 主从不一致
非幂等语句、非确定性函数导致复制偏差

RC + Statement 模式下使用函数如：

  • NOW()
  • UUID()
  • RAND()

因为这些函数在主从执行结果不同！

MVCC（多版本并发控制）

MVCC 是一种通过维护数据的多个版本来实现高并发访问的数据库技术。是保障事务隔离性的核心手段。实现读写并发：读不阻塞写，写不阻塞读。

MVCC的基石, Undo Log: 存储被修改或删除行的旧版本数据，用于构建版本链。

读操作访问历史快照（通过 Undo Log 构造），而非实时数据。

知识补充

事务ID (Transaction ID, XID)

  • 全局唯一递增的事务标识符（32位或64位），用于标识数据版本的创建者和修改者。
  • 事务首次执行修改操作时才会被分配XID（惰性分配）。

InnoDB 行记录结构中与 MVCC 相关的两个隐藏字段

字段名含义
DB_TRX_ID插入该版本的事务ID
DB_ROLL_PTR指向 undo log（旧版本的指针）

旧版本数据存储在 undo 段 中，通过指针形成版本链表。

快照

在 MySQL（InnoDB 引擎） 中，快照（Snapshot） 是 MVCC（多版本并发控制）机制的核心，用于实现 非阻塞的一致性读，
特别是在 REPEATABLE READ 和 READ COMMITTED 隔离级别下。

在 MVCC（多版本并发控制）中，“快照”是一个核心概念，它定义了事务在特定时间点“看到”的数据库的一致视图。
它不是物理复制整个数据库，而是一种逻辑机制，通过元数据和版本链来实现。

读写不阻塞：读操作（使用快照）不需要获取锁来阻塞写操作。写操作创建新版本，不会直接覆盖旧版本，因此不会阻塞正在读取旧版本的读操作。

核心概念总览

元素作用
快照当前事务能“看到”的数据版本集合，是一种一致性视图
undo log存储旧版本数据，用于构造快照
trx_id每个事务的唯一标识
Read View快照的实现载体，事务开启时生成 Read View，记录活跃事务ID

快照的生成时机

事务在 执行第一个查询语句 或 启动后第一次读数据 时生成一个数据快照，定义了该事务可见的数据范围。

InnoDB 在如下两种 SQL 隔离级别中创建快照：

  • REPEATABLE READ：在第一次读取数据时生成快照，整个事务期间保持不变
  • READ COMMITTED：每次读取数据都会生成新的快照（每次都看“已提交的最新版本”）

快照维护者 -> Read View（读取视图）

Read View 是一个包含以下几个字段的数据结构，关键字段如下：

字段名作用说明
m_low_limit_id活跃事务ID集合中的最小值（低水位）
m_up_limit_id当前系统分配的下一个事务ID（即最大事务ID，高水位）
m_creator_trx_id当前创建 ReadView 的事务ID
m_ids当前活跃事务ID集合（未提交）

这些字段共同决定了事务在访问数据库时的视图，视图定义了哪些事务产生的数据是对当前事务可见的。

可见性规则 (Visibility Rules)

数据版本的可见性规则，就是基于数据的 row trx_id 和这个一致性视图的对比结果得到的。

视图把所有的 row trx_id 分成了几种不同的情况。

当当前事务 T 读取一条记录 R 时，会根据 Read View 判断它是否可见：

行版本的 DB_TRX_ID与ReadView比较说明
R.trx_id == m_creator_trx_id自身事务的修改总是可见
R.trx_id > m_creator_trx_id该版本的记录来自未来事务，当前事务不可见。
R.trx_id < m_low_limit_id该版本的记录由一个已经提交的事务创建，对当前事务可见。
R.trx_id in m_ids该版本的记录来自并发未提交的事务，当前事务不可见。

不在活跃事务集合中，且在高水位和低水位之间，可见。

图解 Read View 工作原理

假设系统当前状态：

  • 最新分配的事务 ID：120
  • 活跃事务 (m_ids)：[102, 105, 110]
  • 创建 Read View RV1 的事务 ID (creator_trx_id)：100

则 RV1 的关键属性为：

  • m_low_limit_id = 120 (下一个待分配 ID)
  • m_up_limit_id = 102 (m_ids 中最小值)
  • m_ids = [102, 105, 110]
  • m_creator_trx_id = 100

快照读（Snapshot Read） VS 当前读（Current Read）

快照读（Snapshot Read）

  • 通过 MVCC 读取历史版本数据（如普通 SELECT）
  • 无锁，高性能

当前读（Current Read）

  • 加锁读（如 SELECT … FOR UPDATE）
  • 会阻塞其他事务修改

事务传播行为（Spring 语境）

当一个方法在事务上下文中被调用时，如果调用方已有事务，Spring 应该如何处理被调用方法的事务。

Spring 的 7 种事务传播行为

传播行为（常量）描述简要
REQUIRED（默认）有事务就加入，没有就新建一个事务
REQUIRES_NEW总是新建事务，挂起调用方的事务
NESTED如果已有事务，则在嵌套事务中执行（使用保存点 Savepoint）
SUPPORTS有事务就加入，没有就非事务执行
NOT_SUPPORTED始终非事务执行，有事务则挂起它
MANDATORY必须在事务中运行，否则抛出异常
NEVER必须在无事务的上下文中运行，有事务就抛异常

事务处理

需要知道的几个术语

  • 事务（transaction）: 指一组SQL语句；
  • 回退（rollback）: 指撤销指定SQL语句的过程；
  • 提交（commit）: 指将未存储的SQL语句结果写入数据库表；
  • 保留点（savepoint）: 指事务处理中设置的临时占位符（place-holder），可以对它发布回退（与回退整个事务处理不同）。

begin/start transaction 命令并不是一个事务的起点，在执行到它们之后的第一个操作 InnoDB 表的语句，事务才真正启动。

如果想要马上启动一个事务，可以使用 start transaction with consistent snapshot 这个命令。

锁机制

分类

按锁的粒度 (Lock Granularity)

  • 全局锁: 整个数据库实例的所有表。
  • 表级锁: 整张表。
  • 行级锁 (InnoDB 核心特性): 单行记录（或索引记录）。

按访问权限分

共享锁 (Shared Lock - S Lock / 读锁)

  • 允许其他事务读取锁定的行，但不能修改。
  • 多个事务可以同时持有同一行的 S 锁。
  • SELECT … LOCK IN SHARE MODE 显式获取。SELECT 在 SERIALIZABLE 隔离级别下也可能隐式获取（取决于配置）。

排他锁 (Exclusive Lock - X Lock / 写锁)

  • 阻止其他事务读取（除非是 READ UNCOMMITTED）或修改锁定的行。
  • 一次只能有一个事务持有某一行的 X 锁。
  • INSERT, UPDATE, DELETE 操作会隐式获取受影响行的 X 锁。SELECT … FOR UPDATE 显式获取。

表级锁

  • 表共享读锁
  • 表独占写锁
  • 意向锁 (Intention Lock - InnoDB特有)： 表级锁，表示事务稍后将在表中的行上请求哪种类型的锁（共享或独占）。
  • 自增锁 (AUTO-INC Locks)： 特殊表锁，用于处理具有 AUTO_INCREMENT 列的表上的并发插入。在事务执行插入语句时持有（不是整个事务），确保自增值连续且唯一。
  • 元数据锁 (Metadata Lock - MDL)： MySQL 5.5+引入。保护表结构（schema）。

行级锁

记录锁 (Record Lock)： 锁定索引中的一条具体记录。主键或唯一索引上的等值查询通常加记录锁。

间隙锁 (Gap Lock)： 锁定索引记录之间的间隙（一个开区间），防止其他事务在这个间隙中插入新记录。这是实现 REPEATABLE READ
隔离级别防止幻读的关键机制之一。
仅作用于非唯一索引或范围查询。唯一索引的等值查询（记录存在）不需要间隙锁。

临键锁 (Next-Key Lock)： 记录锁 + 间隙锁的组合。锁定索引记录本身以及该记录之前的间隙（一个左开右闭区间）。
这是 InnoDB 在 REPEATABLE READ 隔离级别下默认的行锁算法。目的是解决幻读。

锁定位置

锁定位置： InnoDB 的行锁是加在索引上的！
即使表没有显式定义索引，InnoDB 也会创建一个隐藏的聚簇索引（通常是主键）。锁是加在索引记录上的。

通过主键或唯一索引进行精确查询，锁只加在满足条件的特定行上。

通过非唯一索引进行查询，除了锁住满足条件的索引记录外，还会锁住相关的间隙（间隙锁或临键锁）。

没有使用索引的查询 (WHERE 条件无索引列)： InnoDB 无法使用行锁，会退化为锁住整个表的所有行（实际上是所有索引记录和间隙）！
相当于表锁的效果，并发性能急剧下降。务必避免！

RR 级别在 MySQL 中通过 Next-Key Lock 完全解决了幻读吗

结论：RR 级别通过 Next-Key Lock 解决了“写操作导致的幻读”，但“读操作幻读”在特定场景下仍可能发生

  • 这取决于操作类型（读 vs 写）和语句执行方式（快照读 vs 当前读）

幻读的本质

  • 指同一事务内多次查询同一范围时，后一次查询看到了前一次查询未出现的行（如新插入的行）。

视图和临时表

对比

特性视图（VIEW）临时表（TEMPORARY TABLE）
定义是一个虚拟表，本质上是预定义的 SELECT 查询结果集是一个真实存在的临时表，数据存储在数据库中
存储不存储数据，只保存查询定义存储数据，实际占用空间
作用域对所有用户可见，只要有权限均可访问会话级别（session）可见，断开连接即自动删除
生命周期持久存在，直到显式 DROP VIEW只在当前数据库连接有效，断开连接即销毁
更新和操作对可更新视图，可以通过视图操作底层表可以对临时表执行任何 DML 操作（增删改查）
创建语法CREATE VIEW view_name AS SELECT ...CREATE TEMPORARY TABLE temp_table AS SELECT ...
性能每次查询时执行视图定义的 SQL，实时获取数据数据已物理存在，查询速度快，但需要额外空间存储数据
使用场景简化复杂查询、权限控制、封装业务逻辑存储中间结果、数据处理临时存储、复杂计算临时存储
索引视图本身不能有索引临时表支持索引，可以优化查询性能
