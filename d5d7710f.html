JIT优化

  • 解释器（Interpreter）：逐行解释执行字节码，启动快但慢。
  • JIT 编译器：热点代码在运行时被编译为本地机器码，提高性能。
  • JIT 并不会编译所有代码，而是采用 热点探测 + 编译 策略

热点代码识别方式

  • 方法调用计数（方法调用次数）
  • 回边计数（Back Edge Counting）：判断循环次数
  • 内联调用深度

JIT 常见优化技术

方法内联 、 逃逸分析 、循环展开、循环优化 、 死代码消除 、 常量折叠、公共子表达式消除

逃逸分析（Escape Analysis）的优化

现代JVM（如HotSpot）在运行时通过逃逸分析判断对象的生命周期是否仅限于当前方法或线程：

若对象未逃逸，JVM可能应用以下优化：

标量替换（Scalar Replacement）
将对象拆解为基本类型（如int、long），直接在栈上分配其字段，避免创建完整对象。

栈上分配（Stack Allocation）
直接将对象分配在栈帧中，随方法结束自动销毁（无需GC介入）。
注：HotSpot JVM主要使用标量替换，栈上分配实际较少使用。

锁消除（Lock Elision）：移除未逃逸对象的同步锁

Java中的对象一定在堆上分配内存吗

在Java中，对象通常（且绝大多数情况下）在堆（Heap）上分配内存，这是Java内存模型的基本规则。

但现代JVM（如HotSpot）会通过_逃逸分析（Escape Analysis）_ 等优化技术，在特定条件下将对象拆解或直接分配在栈（Stack）上，从而避免堆分配的开销。

JVM如何判断对象是否存活

两种算法

引用计数法

  • 每个对象维护一个引用计数器，有引用时加 1，引用失效时减 1，为 0 表示对象可回收。
  • 无法解决循环引用问题，因此 HotSpot JVM 不使用它。

可达性分析法

从“GC Roots”出发，沿着对象引用链向下搜索，能到达的对象都是“存活的”；反之则是“不可达”。

GC Roots 是一组特殊的对象作为根节点，它们永远是“存活的”

来源类型描述
虚拟机栈中的引用方法区中的局部变量表、参数等（即：栈帧中的引用）
方法区中的类静态属性引用的对象static 修饰的引用变量
方法区中常量引用final 修饰的引用，或字符串常量池
JNI 引用（本地方法栈）本地代码（如 C/C++）引用的对象
活动线程对象自身所有正在运行的线程对象

对象不可达 ≠ 马上回收（“两次判死”机制）

JVM 判断对象是否存活的全过程

  • 使用可达性分析法判断引用链是否连接到 GC Roots
  • 如果不可达，查看是否实现 finalize()
  • 如果实现且未执行过，则进入 F-Queue 调用 finalize()
  • finalize 后如果仍不可达，则对象被回收

三色标记法

方面说明
三色分类白（未访问）/灰（待处理）/黑（处理完成）
目的支持并发标记时的正确 GC
问题并发引用变更可能误回收
解决方案写屏障（插入/删除），保持引用快照一致
应用CMS、G1、ZGC、Shenandoah 等现代 GC 收集器

分代收集（Generational Collection）

核心思想：根据对象存活时间划分内存区域，针对性使用算法。

堆为什么分代？

java中大部分对象都是朝生夕死的，同时也会有部分对象持久存在。如果把这两部分对象放到一起分析和回收，效率太低。通过对象存活时间划分内存区域，可以显著改善系统性能。

对象从新生代到老年代的生命周期

Java 堆内存通常被划分为 新生代（Young Generation） 和 老年代（Old Generation），这两部分采用了不同的垃圾回收器和策略，适配各自对象的生命周期特点。

1. 对象创建：分配在新生代 Eden 区

  • 触发条件：new Object()
  • 内存分配：
    • 优先在 Eden 区分配（占用 Eden 空间）。
    • 若 Eden 空间不足，触发 Minor GC。

2. 第一次 Minor GC：存活对象进入 Survivor 区

  • 触发条件：Eden 区满时触发 Minor GC。
  • 复制过程：
    1. 扫描 Eden + Survivor From 区的存活对象。
    2. 将存活对象复制到 Survivor To 区（年龄 Age=1）。
    3. 清空 Eden + Survivor From 区。
  • Survivor 区结构：
    • 两个等大区域（S0 和 S1），每次 GC 后互换角色（From ↔ To）。
  • 对象年龄（Age）：
    • 每熬过一次 Minor GC，年龄 +1（存储在对象头中）。

3. 在 Survivor 区多次存活

  • 重复过程：
    • 每次 Minor GC 后，存活对象在 S0 和 S1 间复制。
    • 年龄持续增加（直到达到阈值）。
  • 动态年龄判定：
    • 若某年龄对象的总大小 > Survivor 区的 50%，则 ≥该年龄的对象直接晋升老年代。
    • _例：Survivor 区 100MB，年龄3的对象总大小 60MB → 所有年龄≥3的对象晋升_。

4. 晋升老年代的条件

满足以下任一条件时，对象进入老年代：

条件说明触发场景
年龄阈值对象年龄 ≥ MaxTenuringThreshold （默认 15）长期存活的缓存对象
大对象直接分配对象大小 > -XX:PretenureSizeThreshold （默认 0，由回收器决定）大数组（如 10MB 的 byte[] ）
动态年龄判定Survivor 区中同年龄对象总大小 > Survivor 区的 50%短期产生大量存活对象
分配担保失败Minor GC 后 Survivor 空间不足，且老年代空间不足，对象直接进入老年代高频创建大对象
Survivor 空间不足Minor GC 时存活对象大小 > Survivor 区容量未达年龄阈值但 Survivor 已满

5. 老年代的对象生命周期

  • 存储位置：老年代（Old Generation）
  • 回收机制：
    • Major GC/Full GC：扫描整个堆（新生代+老年代+元空间），耗时长（秒级 STW）。
    • 触发条件：老年代空间不足（java.lang.OutOfMemoryError: Java heap space）。
  • 回收算法：
    • CMS：并发标记清除（避免长时间 STW）。
    • G1/ZGC：并发标记整理（低延迟）。
  • 终结：
    • 对象无引用时被标记为垃圾，GC 回收内存。

Major GC 和 Full GC

在 Java 垃圾回收中，Major GC 和 Full GC 是两个容易混淆但实际有本质区别的概念。以下是清晰对比和深度解析：

特性Major GCFull GC
回收范围仅老年代整个堆 + 元空间
触发条件老年代空间不足堆/元空间不足、System.gc() 等
STW 时间通常较短（CMS 并发）较长（秒级，整堆回收）
回收器行为取决于回收器（如 CMS 并发）所有回收器均暂停所有线程
关联性可能单独发生可能包含 Major GC + Minor GC

Major GC：老年代专属回收

定义

  • 仅回收老年代（Old Generation），不涉及新生代（Young Generation）和元空间（Metaspace）。
  • 是 分代模型下的特定行为（如 CMS、Parallel Old 等回收器）。

触发条件

  1. 老年代空间不足（例如：晋升对象过多、大对象直接分配）。
  2. 分配担保失败（Minor GC 前预测老年代空间不足）。

Full GC：整堆回收的“终极手段”

定义

  • 回收整个堆（新生代 + 老年代）+ 元空间。
  • 全局 STW（Stop-The-World），暂停所有应用线程。

触发条件

  1. 老年代空间不足：Major GC 后仍无法满足分配。
  2. 元空间不足：类加载过多（如动态代理滥用）。
  3. 显式调用：System.gc()（可通过 -XX:+DisableExplicitGC 禁用）。
  4. 分配担保失败：Minor GC 后 Survivor 放不下，老年代也放不下。
  5. Heap Dump 操作：手动或 OOM 时的快照生成。

回收器行为对比

回收器Full GC 行为STW 时间
Serial GC标记-整理整个堆长（秒级）
Parallel GC多线程标记-整理（吞吐优先）较长
CMS退化为 Serial Old（单线程标记-整理）→ 灾难性停顿极长（10s+）
G1退化为单线程 Serial Old（JDK 10 前）→ 应极力避免极长
ZGC/Shenandoah无 Full GC 概念（全程并发回收）<1ms（无 STW）

💡 关键点：CMS/G1 的 Full GC 是故障回退机制，表明并发回收失败。

内存泄漏（Memory Leak）和内存溢（OutOfMemory）

是两种不同的内存问题，本质区别在于：泄漏是内存无法回收的慢性病，溢出是内存不足的急性崩溃。

内存泄漏（Memory Leak）

定义

对象不再被使用，但GC无法回收，导致内存被无效占用。

根本原因

  • 对象被意外持有：
    • 静态集合类（如 static Map）持续添加元素未清除。
    • 未关闭资源（Connection、Stream）导致关联对象无法回收。
    • 监听器/回调未注销（如事件监听器）。
  • 内部类隐式引用：非静态内部类持有外部类引用（如Handler导致Activity泄漏）。
  • 缓存未清理：缓存策略缺失（如无LRU淘汰机制）。

特征

  • 症状隐蔽：进程内存缓慢增长（如每天增长1%）。
  • 长期影响：持续消耗内存，最终触发内存溢出。

  • jmap -histo:live <pid>：查看对象实例数。

内存溢出（OutOfMemory, OOM）

定义

JVM 内存耗尽，无法分配新对象，触发 java.lang.OutOfMemoryError。

直接原因

  • 堆空间不足：
    • 内存泄漏积累到临界点。
    • 高并发场景突发大对象分配（如报表导出）。
  • 元空间不足：动态生成类过多（如CGLib代理）。
  • 线程栈溢出：-Xss 设置过小或递归过深（StackOverflowError是OOM子类）。
  • 直接内存不足：NIO的 ByteBuffer.allocateDirect() 超出 -XX:MaxDirectMemorySize。

特征

  • 突发崩溃：服务瞬间不可用。
  • 明确错误：JVM 抛出 OutOfMemoryError。

在使用 Eclipse MAT（Memory Analyzer Tool） 分析 Java 堆内存时，两个最关键的概念是：

🌱 1. Shallow Heap（浅堆大小）

表示 一个对象本身占用的内存大小，不包括它引用的其它对象。

  • 通常包括对象头（对象元信息）+ 基本字段（如 int、long 等）+ 引用字段（仅引用地址）。
  • 例如：
    • 一个 Object 占 16 bytes（在 64 位 JVM 开启压缩指针时）。
    • 一个 User 对象有 3 个字段（2 个基本类型 + 1 个引用），其 shallow heap 可能是 24 bytes。

📌 类似于：

“我自己身体有多重，不管我背了几个人。”

🌳 2. Retained Heap（保留堆大小）

表示 某个对象被 GC Root 保留时，会连带保留下来的所有对象所占内存总和。

  • 换句话说：
    • 如果这个对象被 GC 回收掉，它所能引用到的所有对象（即依赖它活着的）也会被一并回收，这部分内存称为其 Retained Heap。
  • Retained Heap = Shallow Heap + 所有无法独立存活的下游对象的 Shallow Heap 总和

📌 类似于：

“我如果死了，还会拖死多少人？”

判断两个类是否为“同一个类”的标准

条件说明
类的完全限定名（Fully Qualified Name）相同即包名 + 类名一致，例如 com.example.MyClass
由同一个类加载器（ClassLoader）加载即使类名一致，如果是由不同的类加载器加载，也视为不同类
类签名一致字段、方法等签名一致（编译后的 class 结构兼容）

类的生命周期 & 类加载过程

在 JVM 中，类的生命周期是指一个类从被加载进内存、到被使用、最后被卸载的整个过程。这个过程由 JVM 管理，主要涉及以下7个阶段

阶段简要说明
1️⃣ 加载（Loading）+ 通过类加载器（ClassLoader）读取 .class 文件并转为二进制数据，生成 Class 对象。 + 不一定从磁盘加载，也可能来自网络、数据库、内存中（自定义类加载器）
2️⃣ 验证（Verification）校验字节码的合法性、安全性、格式等
3️⃣ 准备（Preparation）+ 为 static 字段分配内存，设置为默认值 + static int count = 10; 此时只是 count=0
4️⃣ 解析（Resolution）把符号引用（常量池中的类、方法、字段）转为直接引用
5️⃣ 初始化（Initialization）+ 执行静态代码块、静态字段的初始化赋值。 + 线程安全：JVM 加锁保证只执行一次
6️⃣ 使用（Using）调用类方法、访问静态变量/字段、创建实例、反射操作时，类才真正被“用”起来。
7️⃣ 卸载（Unloading）1. 类所有实例已被 GC 2. 加载该类的 ClassLoader 已被 GC 3. 类对应的 Class 对象无任何引用（包括反射）

在 Java 中，类加载过程是指 JVM 将 .class 文件加载到内存，并将其转化为Class对象的全过程。这是类生命周期的前半部分，主要涉及前5个步骤

类何时被初始化？

JVM 规定了几种“主动使用”会触发初始化：

触发行为是否初始化
访问静态字段（非 final）✅ 是
调用类的静态方法✅ 是
使用 Class.forName() 加载类✅ 是
子类初始化，父类也会先初始化✅ 是
new 一个类对象✅ 是
数组引用类（如 A[] arr = new A[10]）❌ 否
获取 Class 对象（A.class）❌ 否

什么是符号引用（Symbolic Reference）？

在 .class 字节码中，类、字段、方法的引用 不是用内存地址表示的，而是用一种逻辑名称表示，叫做符号引用。

在编译后的字节码中：

  • User 是类的 全限定名字符串：com/example/User
  • getName 是方法名和描述符：getName()Ljava/lang/String;

它们以字符串形式保存在 class 文件的 常量池 中（Constant Pool），这就是“符号引用”。

什么是直接引用（Direct Reference）？

解析完成后，JVM 将这些符号引用解析为直接引用：

直接引用是指向类或成员在内存中具体地址或结构的引用，通常是指向方法区中类结构的指针。

比如：

  • 指向已加载类的 Class 对象的引用
  • 方法或字段的偏移量、内存地址等

这样以后执行方法调用、访问字段时就可以直接定位，提高性能。

为什么要做解析？

JVM 在类加载时不能立即将所有依赖类都加载进来（性能、懒加载考虑），所以符号引用是必须的。而在实际运行时：

  • 需要将“字符串描述的类名” → 解析成内存中的 Class 对象
  • 方法/字段名 → 变为实际的方法地址或方法表索引

否则 JVM 无法执行这些调用。

类加载器与双亲委派

如何破坏双亲委派

虽然双亲委派是默认机制，但在某些特定场景中必须打破它，例如：

  • 动态加载插件（每个插件有独立的类）
  • 热部署（新类覆盖旧类）
  • 加载第三方库中版本冲突的类

破坏方式一：重写 loadClass() 方法

可以通过对某些类指定前缀，阻止委派加载，实现“选择性破坏”。

破坏方式二：用 URLClassLoader 自定义加载路径

这样加载器没有父类，任何类都靠它自己加载。

破坏方式三：SPI 加载机制中的破坏

Java SPI（ServiceLoader）加载服务时会使用线程上下文类加载器 Thread.currentThread().getContextClassLoader()，而非
AppClassLoader，从而避免双亲委派，加载用户自定义实现类。

方法区、永久代、元空间

方法区（Method Area）—— JVM 规范定义

  • 定义：
    方法区是 JVM 规范中定义的一个逻辑内存区域（类似“接口”），用于存储：
    • 类的元数据（Class 元信息：类名、字段、方法、常量池等）
    • 静态变量（static）
    • JIT 编译后的代码缓存
    • 运行时常量池（Runtime Constant Pool）
  • 特点：
    • 所有线程共享。
    • 是 规范层面的概念，不规定具体实现方式。

永久代（Permanent Generation）—— HotSpot 的具体实现

  • 定义：
    永久代是 HotSpot 虚拟机（JDK 8 之前）对方法区的具体实现（类似“实现类”）。

元空间（Metaspace）—— JDK 8 及之后的实现

  • 取代永久代：JDK 8 起，HotSpot 移除了永久代，改用 元空间（Metaspace） 作为方法区的实现。
  • 优势：
    • 避免 OOM：本地内存理论上只受系统资源限制。
    • 自动扩容：无需手动设置大小（除非主动限制）。
    • 垃圾回收优化：由元空间自己的垃圾回收器处理，与堆 GC 解耦。

核心变化

特性永久代元空间
存储位置JVM 堆内存中本地内存（Native Memory）
内存管理JVM 管理操作系统自动管理
溢出错误OutOfMemoryError: PermGenOutOfMemoryError: Metaspace
大小限制固定大小（需手动调优）默认无上限（受物理内存限制）
调优参数-XX:PermSize / MaxPermSize-XX:MetaspaceSize / MaxMetaspaceSize

强引用、软引用、弱引用和虚引用

在 Java 中，引用类型分为四种：强引用（StrongReference）、软引用（SoftReference）、弱引用（WeakReference） 和
虚引用（PhantomReference）_。它们体现了对象被垃圾回收（GC）时的__可达性强弱_，对内存管理、缓存设计、资源释放等场景非常关键。

引用类型回收条件常见用途是否影响 GC 回收是否可通过引用访问对象
强引用永远不会被 GC 主动回收（除非引用断开）普通对象引用✅ 无法回收✅ 可以
软引用内存不足时 被 GC 回收缓存（如图片、数据）⚠️ 低内存时回收✅ 可以
弱引用下一次 GC 就会被回收ThreadLocal、类缓存清理等✅ 一定回收✅ 可以（短暂）
虚引用被创建后立即会被 GC 回收，无 get() 方法跟踪对象回收、清理资源✅ 一定回收❌ 不可以（get 返回 null）

常见启动参数解析

内存区域大小相关

-Xms、-Xmx

分别设置 JVM 堆内存的初始大小 (-Xms) 和 最大大小 (-Xmx)。

  • 设置相同值置相同值： (强烈推荐) 避免堆在运行时动态扩展，防止扩展操作带来的额外GC停顿。

  • 通常建议不超过物理内存的50%-70%，留给操作系统和其他进程。

  • 通过监控工具 (jstat, VisualVM, JMC, Prometheus+Grafana) 观察应用稳定运行后（尤其是经过压测后）的 老年代使用量 和 Full GC
    后的堆使用量。

  • -Xmx 应至少比观察到的 老年代稳定使用量 高出 20%-30%，以容纳临时对象和避免频繁 Full GC。-Xms 直接等于 -Xmx。

-Xmn / -XX:NewSize 和 -XX:MaxNewSize

  • Xmn: 同时设置年轻代的初始和最大大小（推荐方式）。
  • XX:NewSize 和 -XX:MaxNewSize: 分别设置初始和最大年轻代大小（通常设置相同值）。

-Xmn显式指定： (推荐) 当你知道应用的对象分配速率和生命周期特点时，优先使用。

较大的年轻代：适合大量短生命周期对象的应用 (-Xmn 接近 -Xmx 的 1/3 到 1/2) 。

  • 好处：减少对象过早晋升到老年代，降低 Full GC 频率。
  • 坏处：单次 Minor GC 时间可能变长（需要复制的存活对象更多）。

较小的年轻代：适合对象生命周期较长或对 Minor GC 停顿极其敏感的应用 (-Xmn 接近 -Xmx 的 1/4 或更小)

  • 好处：单次 Minor GC 时间短。
  • 坏处：对象晋升老年代更快，增加 Full GC 频率。

如何确定大小？ 监控年轻代 GC 频率、晋升到老年代的对象速率。目标是让绝大多数对象在年轻代的 Minor GC 中就被回收掉，避免频繁Full
GC。

同时要保证 Minor GC 的停顿时间在可接受范围内（通常几十到几百毫秒）。

-XX:NewRatio

是 JVM 中用于控制 老年代（Old Generation）与新生代（Young Generation）大小比例 的参数，主要用于整体堆内存结构的分布优化。

-XX:NewRatio=N 表示：老年代与新生代的内存比例是 N:1。 默认值通常是 2（老年代：年轻代=2:1）。

假设：

  • JVM 堆总大小为 -Xms = -Xmx = 800MB
  • 设置了：-XX:NewRatio=2
    则：
  • 新生代（Young Gen）占 800 / (2+1) = 266MB
  • 老年代（Old Gen）占 800 - 266 = 534MB

-XX:SurvivorRatio

设置 Eden 区与一个 Survivor 区的比例

表示 Eden:S0:S1 = 8:1:1

  • 新生代总共被分为 8+1+1 = 10 份；
  • Eden 占 8/10；
  • 每个 Survivor 区占 1/10。

增大 Eden

  • 适合对象分配速率极高且绝大多数对象都是朝生夕死的应用，减少 Minor GC 频率。
  • 风险是如果 Survivor 相对太小，Minor GC 后存活的对象可能无法完全放入 Survivor，会直接晋升到老年代。

减小 Eden

  • 适合对象存活率较高的应用。给 Survivor 更多空间，让对象能在 Survivor 区经历更多次 GC 才晋升，减少不必要的晋升。
  • 坏处是 Minor GC 会更频繁（因为 Eden 小了）

例如

  • Survivor 区经常接近满 (>90%): 增大 Survivor (减小 SurvivorRatio 值)

  • 在 Minor GC 后发现有对象因为 Survivor 空间不足直接晋升: 增大年轻代 (-Xmn)

-XX:MetaspaceSize 和 -XX:MaxMetaspaceSize (Java 8+)

设置 元空间 (Metaspace) 的初始大小 (-XX:MetaspaceSize) 和最大大小 (-XX:MaxMetaspaceSize)。

  • 元空间存储类元数据信息（取代了 Java 8 之前的永久代 PermGen）。

  • -XX:MaxMetaspaceSize: 强烈建议在生产环境设置！ 避免元空间无限增长导致系统内存耗尽。取值根据应用加载的类数量确定。

  • 对于大型应用或频繁动态生成类（如 Groovy, Spring AOP 代理）的应用，可能需要设置较大值（如 256m, 512m, 1g）。

监控元空间使用量 (jstat -gcmetacapacity) 是确定合理上限的关键。

垃圾收集器选择与相关参数

CMS 收集器关键参数

-XX:+UseParNewGC + -XX:+UseConcMarkSweepGC

ParNew 负责年轻代 (并行复制)，CMS 负责老年代（并发标记清除）。核心目标是降低老年代 GC 的停顿时间。适用于Web应用、服务端等对响应时间敏感的场景。

注意：CMS 在 JDK 9 被标记为 Deprecated，在 JDK 14 中被移除。

-XX:CMSInitiatingOccupancyFraction

设置老年代空间使用率达到多少百分比时，开始 CMS 并发收集周期。

必须配合 -XX:+UseCMSInitiatingOccupancyOnly: 确保仅根据老年代占用率触发 CMS，而不是 JVM 自行判断的其他启发式规则。

场景与取值

默认值 (68)： 比较保守。

  • 调高 (70-80)

如果应用老年代增长缓慢且 Full GC 频率很低，可以调高以延迟 CMS 启动，减少并发收集的开销。
如果老年代增长过快，在 CMS 完成前就满了，会触发“Concurrent Mode Failure”，退化为 Serial Old GC 进行 Full GC，停顿时间很长。

  • 调低 (60-70)

如果老年代增长很快或经常发生 “Concurrent Mode Failure”，需要调低此值，让 CMS 更早启动，预留更多空间给浮动垃圾。
坏处： CMS 启动更频繁，并发收集开销增加。

G1 收集器关键参数

-XX:MaxGCPauseMillis

期望达到的最大 GC 停顿时间目标（毫秒）。这不是一个硬性保证，而是 G1 努力的方向。G1 会根据这个目标自动调整其他参数（如每次回收的
Region 数量）。

场景与取值：

默认值 (200ms)： 适用于大多数对延迟有一定要求但不是极致的场景。

  • 调低 (50-150ms)

对延迟要求更高的应用。G1会尝试做更多工作（如更早开始回收、每次回收更少的Region）来满足更短的停顿目标。可能导致GC频率增加，总体吞吐量下降。

  • 调高 (200-500ms)

对吞吐量更敏感，可以容忍稍长停顿的应用。G1 会减少 GC 开销，提升吞吐量。

根据应用的 SLA (服务等级协议) 要求设定。需要监控实际停顿时间 (-Xlog:gc* 或 jstat -gc) 是否达标。

-XX:InitiatingHeapOccupancyPercent

设置整个堆的使用率达到多少百分比时，开始并发标记周期。

场景与取值： 类似 CMS 的 CMSInitiatingOccupancyFraction，但针对整个堆。

默认值 (45)： 比较保守。

  • 调高 (50-60)

如果堆使用率增长缓慢且混合GC能有效回收，可以调高以减少并发标记开销。坏处：如果堆增长过快，在标记完成前空间不足，会触发FullGC。

  • 调低 (35-45)

如果堆增长很快或经常发生 Full GC (尤其是因为分配失败引发的)，需要调低此值让并发标记更早开始。坏处：并发标记更频繁，开销增加。
