基础查询

SQL语句执行顺序

  • WITH、FROM、ON 、JOIN、WHERE、GROUP BY、AGG_FUNC、HAVING、SELECT、UNION、DISTINCT 、ORDER BY、LIMIT。
  • FROM、ON 、JOIN、WHERE、GROUP BY、AGG_FUNC、HAVING、SELECT、DISTINCT 、ORDER BY、LIMIT。

SQL 语句执行的逻辑顺序：（自上而下）

关键字作用说明
WITH（CTE）公共表达式（子查询临时结果集），为主查询提供中间视图。
FROM确定查询数据的表或子查询来源。
JOIN / ON若有连接操作，执行连接并应用连接条件（ON）筛选。
WHERE对原始表或连接结果进行行级过滤。
GROUP BY将符合 WHERE 的结果进行分组。
聚合函数 SUM(), COUNT() 等在分组后的基础上，计算聚合值。
HAVING对分组结果进行过滤（区别于 WHERE 过滤原始行）。
SELECT确定最终返回的列，包括聚合结果或字段。
DISTINCT对 SELECT 出来的结果进行去重。
UNION / UNION ALL若有多个查询结果集合并，执行合并。
ORDER BY对最终结果进行排序。
LIMIT / OFFSET限制返回结果的条数或分页。

案例

去重（DISTINCT）

用于去重的关键字，常用于 SELECT 查询中，消除结果集中重复的记录。

  • 作用：返回结果中 column1, column2, … 组合唯一的行（重复的行只保留一条）。
  • 位置：DISTINCT 紧跟在 SELECT 后。
  • 作用范围：作用于 SELECT 字段组合，而不是单个字段。

用法

总结对比表

对比项IN (SELECT)EXISTS (SELECT)
子查询类型非相关子查询（物化执行）相关子查询（每行动态执行）
去重自动去重（逻辑层）不去重（存在一条即返回 TRUE）
临时表会构建临时表不构建
索引使用子查询可优化走索引主表+子表都能利用索引
数据量小时优势快（整体构造临时集合后快速比对）稍慢（每行查子表）
数据量大时优势慢（物化+大集合查找性能差）快（存在即返回，避免全查）
语义“是否属于集合”“是否存在某种匹配”
推荐使用场景子查询数据较小、字段唯一性强子查询结果大、重复多、存在性判断为主

注意

子查询内排序不会对外层生效

如果在子查询里使用了 ORDER BY，但 外层查询没有 ORDER BY 的话，最终结果并不会保证是“排序后的”。

这是因为 SQL 的“结果集顺序”只有在最外层 ORDER BY 明确指定时才有保证。

联结表（join）

自然联结

  • 笛卡儿积（cartesianproduct） : 没有联结条件的表关系返回的结果为笛卡儿积。检索出的行的数目将是第一个表中的行数乘以第二个表中的行数。

内部联结

  • 每个内部联结都是自然联结, 避免缺失连接条件（如 g1.supplier = g2.supplier），导致产生 笛卡尔积

外部联结

与内部联结关联两个表中的行不同的是，外部联结还包括没有关联行的行。在使用OUTER JOIN 语法时，必须使用RIGHT 或LEFT

sql-join 图

假设有两个表：

表 A（左表）

表 B（右表）

每种 JOIN 类型表示的是：从 A 和 B 中选出哪些记录。

JOIN 类型描述图示概念（A 和 B 的集合关系）
INNER JOIN只保留两个表中都存在的匹配记录A ∩ B（交集）
LEFT JOIN保留左表所有记录，右表没有匹配则为 NULLA ∪ (A ∩ B)，即 A 全部 + 匹配 B 的部分
RIGHT JOIN保留右表所有记录，左表没有匹配则为 NULLB ∪ (A ∩ B)，即 B 全部 + 匹配 A 的部分
FULL OUTER JOINA 和 B 所有记录，没有匹配的补 NULLA ∪ B（并集）
LEFT SEMI JOIN保留 A 中“有匹配”的记录，不显示 B 的字段A 中能匹配到 B 的部分
LEFT ANTI JOIN保留 A 中“没有匹配”的记录A 中不能匹配到 B 的部分
CROSS JOIN笛卡尔积：A 的每行都与 B 的每行组合A × B（每行两两组合）
  • MySQL 不支持 FULL OUTER JOIN，可以用 UNION 模拟；
  • SEMI / ANTI JOIN 是逻辑概念，MySQL 会通过 EXISTS 或 NOT EXISTS 来模拟。

分析

使用子查询
  • 子查询: 使用 key_category_id 索引过滤 category_id=2 的供应商（rows=2），并物化为临时表，效率较高。
  • 主查询 g1 对 goods 表进行全表扫描（type=ALL），因为 supplier 字段无索引，导致逐行过滤（Using where）对每行做 IN
    判断（在临时表中是否存在该 supplier）。

✅ 优势

  • 使用了子查询物化（MATERIALIZED），子查询只执行一次，效率比嵌套子查询好。
  • 临时表自动优化去重
  • 结构清晰，业务含义直观。

❌ 劣势

  • 主查询全表扫描 (type = ALL)，因为 supplier 无索引，全表扫描在大数据量时性能差。
  • 物化临时表有内存/磁盘开销
  • IN (…) 子查询较大时，判断效率下降（虽然 MySQL 内部会优化，但不如 JOIN 高效）。

优化建议

创建必要索引

总结

建议说明
✅ 使用自联结代替子查询性能更优，JOIN 可走索引
✅ 添加 supplier 索引无此索引时无法避免全表扫描
✅ 子查询适合数据量小场景大数据量建议使用 JOIN 替代
✅ 避免笛卡尔积写法联结必须使用连接条件如 g1.supplier = g2.supplier
  • 子查询适合小结果集过滤，如 IN (10个以内)
  • 自联结适合“基于属性等值查找”，如 supplier 作为连接字段时配合索引使用。
  • 如有性能问题，考虑写法变更为 JOIN + DISTINCT，避免重复结果。

组合查询

使用UNION

注意：

  1. 在使用UNION 时，重复的行被自动取消。
  2. 如果想返回所有匹配行，可使用UNION ALL。
  3. UNION 中的每个查询必须包含相同的列、表达式或聚集函数（不过各个列不需要以相同的次序列出）。
  4. 列数据类型必须兼容：类型不必完全相同，但必须是DBMS可以隐含地转换的类型（例如，不同的数值类型或不同的日期类型）。
  5. 对组合查询结果排序，只能使用一条ORDER BY 子句，它必须出现在最后一条SELECT语句之后。
