源码属性

AbstractList中的modCount字段阐述

此列表被结构修改的次数。结构修改是指改变列表的大小，或者以某种方式扰乱列表，从而导致正在进行的迭代可能产生不正确的结果。

该字段由迭代器和listIterator方法返回的迭代器和列表迭代器实现使用。

如果该字段的值意外更改，迭代器(或列表迭代器)将抛出ConcurrentModificationException异常，以响应next、remove、previous、set或add操作。

这提供了快速故障行为，而不是在迭代期间面对并发修改时的非确定性行为。

子类是否使用此字段是可选的。如果一个子类希望提供快速失败迭代器(和列表迭代器)，那么它只需要在它的add(int, E)和remove(int)方法(以及它覆盖的导致列表结构修改的任何其他方法)中增加这个字段。

单个调用add(int, E)或remove(int)必须向该字段添加不超过一个，否则迭代器(和列表迭代器)将抛出虚假的concurrentmodificationexception异常。

如果实现不希望提供快速失败迭代器，则可以忽略此字段。

逻辑分析

大致可分为几个步骤：

  1. 入参判空
  2. 初始化removeCount: 方便后面对并发修改异常判断
  3. BitSet：位图
  4. expectedModCount ：预期修改的数量，modCount为ArrayList的属性继承自AbstractList，本文末尾会对modCount字段进行翻译
  5. 遍历符合remove条件的元素存入BitSet
  6. 判断是否并发修改异常（ConcurrentModificationException）
  7. 将幸存的元素移到被删除的元素所留下的空间上 [借助BitSet]，gc回收（将newSize后面的元素手动置空，消除引用），判断是否并发修改异常
  8. removeCount > 0返回结果

可以看出，实际发挥作用的是一个Bitset，有兴趣的小伙伴可以去了解下，这里仅做简单介绍【埋坑】！

Bitset

Bitset，也就是位图，由于可以用非常紧凑的格式来表示给定范围的连续数据而经常出现在各种算法设计中。

基本原理是，用1位来表示一个数据是否出现过，0为没有出现过，1表示出现过。使用用的时候既可根据某一个是否为0表示此数是否出现过。一个1G的空间，有 8102410241024=8.5810^9bit，也就是可以表示85亿个不同的数。常见的应用是那些需要对海量数据进行一些统计工作的时候，比如日志分析等。

ArrayList扩容的执行流程

对JDK源码断点追踪后，梳理了扩容流程：

扩容分为三步：确保内部容量、赋值、返回true

确保内部容量 {@link ArrayList#ensureCapacityInternal(int)}可进一步分解：

  • 计算最小容量{@link ArrayList#calculateCapacity(Object[], int)}，若当前数组无数据，则设置默认长度为10，否则为minCapacity
  • 确保显式容量{@link ArrayList#ensureExplicitCapacity(int)} 关键步骤，做了modCount++及{@link ArrayList#grow(int)} 当minCapacity大于elementData.length时，会触发扩容操作

扩容操作

  1. 按照oldCapacity = elementData.length位运算 右移 (x + x/2) 不等于1.5倍，获得新的容量newCapacity

  2. 新容量与minCapacity比较，取最大值为新容量，且最值要保证在Integer.MAX 与 0 (over flow)之间

  3. 根据新容量大小，拷贝原数据至新数组（新容量大小为新数组长度）

避坑指南

ArrayList并发add问题

  1. 数据丢失问题
  2. 值为null

在ArrayList#add(Object)源码中涉及两个重要参数modCount和size。由于add方法内没有对modCount做并发修改校验,
所以不会触发fail-fast；而size, 它是非volatile，所以不保证可读性（故千万不能用size来判断一个并发add操作是否完成）。
根据以上描述，我们可以用一行代码elementData[size++] = e;解释上面的两个问题：

  • 存在非原子操作size++ 【故存在数据丢失问题】
  • e可能存在指令重排， 在刚刚申请内存空间并绑定内存地址却未赋值时，就被其他线程拿去添加数据了

ArrayList迭代器remove 及 for循环remove

先说结论，要使用迭代器来进行remove操作，for循环remove会报错

迭代器是如何保证循环删除不报错的？

在ArrayList#iterator()内部实例化自己重写的迭代器操作,我们可以看到ArrayList.Itr在Itr类中有句代码
int expectedModCount = modCount;,当执行迭代器内相关方法时会检查两值的变化，若不相等则触发并发修改异常
java.util.ConcurrentModificationException。

在Itr类重写的相关方法中，我们会看到手动维护两值相等的语句{@code expectedModCount = modCount;}。 通过这个细节，我们可以去{@link ArrayList#remove(Object)}源码中寻找踪迹，该方法调用了ArrayList#fastRemove()方法，fastRemove方法中只对modCount++;做了操作，在迭代器中的{@code expectedModCount}就不会与之相等，进而触发并发修改异常（fail-fast）。
参照：ArrayList#removeIf(Predicate) 也维护了final int expectedModCount = modCount;
